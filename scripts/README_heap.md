# Анализ Heap Snapshot (Chrome DevTools)

Скрипты для разбора `.heapsnapshot` и поиска причин «тяжести» и тормозов сайта.

## Файлы

- **`heap_snapshot_parser.py`** — парсер формата Chrome/V8: загрузка JSON, обход узлов, агрегация по типу и по имени конструктора.
- **`analyze_heap.py`** — анализ одного снимка: объём памяти, топ по типам, по конструкторам, по строкам, признаки тяжести.
- **`compare_heaps.py`** — сравнение «до» и «после»: прирост узлов и памяти, главные виновники роста.

## Запуск

```bash
# Анализ одного снимка
python3 scripts/analyze_heap.py "path/to/file.heapsnapshot"

# Сравнение до/после (пути с двоеточием в имени поддерживаются)
python3 scripts/compare_heaps.py "path/to/before.heapsnapshot" "path/to/after.heapsnapshot"
```

Пример с твоими файлами:

```bash
python3 scripts/analyze_heap.py "/Users/nikolai/Downloads/Heap-20260304T223023:before.heapsnapshot"
python3 scripts/compare_heaps.py "/Users/nikolai/Downloads/Heap-20260304T223023:before.heapsnapshot" "/Users/nikolai/Downloads/Heap-20260304T223130:after.heapsnapshot"
```

| Узлов | 119 081 | 152 080 | **+32 999** |
| Self-size | 5.64 MB | 7.17 MB | **+1.54 MB** |

### Что сильнее всего выросло и из-за чего сайт кажется тяжёлым

1. **PerformanceEventTiming (+2742 шт., +328 KB)**  
   Браузер копит события Performance Observer (клики, ввод и т.д.). Если подписка не отключается или буфер не чистится — объекты накапливаются и дают лишнюю память и работу при каждом событии.

2. **Code / closure / native**  
   Больше скомпилированного кода и замыканий — типично после дополнительной инициализации скриптов, виджетов, аналитики.

3. **CSSStyleDeclaration (+1443 шт.)**  
   Рост числа объектов стилей — часто из-за динамического DOM или скриптов, которые читают/меняют `element.style` или `getComputedStyle`.

4. **Object / Array / (object elements)**  
   Общий рост обычных объектов и массивов — возможно кеши, накопление данных в скриптах или в расширениях.

5. **LayoutShift, LayoutShiftAttribution, DOMRectReadOnly**  
   Объекты, связанные с метриками (CLS, layout) и Performance Observer — подтверждают активное использование Performance API без очистки или с большим буфером.

### Рекомендации

- **Performance Observer:**  
  Проверить все вызовы `performance.observe(...)` и `new PerformanceObserver(...)`. По возможности отключать observer при ненадобности (`disconnect()`), ограничивать размер буфера и не держать ссылки на старые записи.

- **Стили и DOM:**  
  Уменьшить частоту обращений к `getComputedStyle` / `element.style` и объём динамически создаваемых/пересчитываемых стилей.

- **Общее накопление:**  
  Рост на ~33k объектов между двумя снимками указывает на возможные утечки или кеши без ограничения размера. Имеет смысл искать подписки (события, observers, таймеры), которые не снимаются при смене страницы или при закрытии компонентов.

- **Один большой объект в памяти:**  
  В обоих снимках есть строка вида `[{"type":"mutation","data":...}]` (~350 KB) — вероятно данные мутаций (например, от отладчика или скрипта наблюдения за DOM). В продакшене такой объект не должен накапливаться.

Зависимостей у скриптов нет: только стандартная библиотека Python 3.
